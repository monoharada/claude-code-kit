# CHECKLIST.md — 情報抽出精度チェック

このドキュメントは `meeting-minutes-pro` スキルの精度チェック層。成果物を出力する前後で精度チェックを実施する。

---

## 1. 固有名詞確認表

VTT の音声認識で揺れやすい社名／ツール名／人名は、**プロジェクト固有の誤記対応表**を参照してクロスチェックする：

- `<project-root>/skills-data/meeting-minutes-pro/vendor_names.md`

このファイルには「VTT で出やすい音写」「正式表記」「誤記例」の 3 列表が収録されており、新しい事業者・ツールが登場した場合は随時追記する。

### 運用

1. 議事録出力直前に、登場する固有名詞を `vendor_names.md` と照合
2. 不確実な固有名詞は `カタカナ表記（要確認）` で明示してエスケープ（RULES.R5）
3. `vendor_names.md` に載っていない固有名詞が VTT に複数回登場した場合、要確認フラグを立てた上でユーザーに `vendor_names.md` への追加を提案する

## 2. VTT 発言者ラベル同一化検出

会議室共有マイク由来で、VTT の発言者ラベルが全員同じになる（例：全て特定アカウント＋連番になる）事象があり、アクション担当者の判定を誤るリスクがある。

### 検出ロジック

1. VTT 内のユニーク発言者ラベル数を数える
2. 参加者数（ユーザーが明示、または `standup_extract.md` の組織セクションから推定）と比較
3. 発言者数が参加者数の 50% 未満なら `speaker_label_collapsed` フラグを立てる
4. フラグ立ち時は、`next_actions[].owner` を `要確認` にしてユーザー承認を求める

### フラグ立ち時の候補出力例

```jsonl
{
  "type": "next_action",
  "task": "<task 内容>",
  "owner": "要確認（VTT発言者ラベル崩壊のため）",
  "due": "YYYY-MM-DD",
  "requires_review": true
}
```

## 3. MP4 ピンポイント参照

VTT のみでは音声表現ニュアンスや画面表示された設計ツール上の要素が追えないため、必要な箇所で MP4 のタイムコードを記録する。

### 参照すべき場面

- 設計ツール上でのリリース判断（どのバリエーションを publish したか）
- 数値（例：コントラスト比 `3:1`、サイズ `48px` 等）が画面表示されたとき
- ホワイトボード／Miro の簡易スケッチ説明

### 参照方法

- `source_refs[].timecode` に `HH:MM:SS` で残す
- 議事録本文から該当箇所を参照したい場合は `（動画 00:23:15 を参照）` のように明示
- MP4 自体は保存せず、パスだけ記録（自動書込はしない）

## 4. PII 赤旗リスト

以下の情報は抽出時に赤旗検出し、出力前に `[REDACTED]` に置換またはユーザー確認を促す：

### 高リスク（自動マスク）

- **メールアドレス**：`\b[\w.+-]+@[\w.-]+\.\w+\b` を検出、`[email_redacted]` に置換
- **電話番号**：日本の携帯・固定電話パターンを検出、`[phone_redacted]` に置換
- **住所**：`〒\d{3}-\d{4}` や都道府県+市区町村+番地を検出
- **マイナンバー／番号**：`\b\d{12}\b` / `\b\d{4}-\d{4}-\d{4}\b` を検出

### 中リスク（要確認フラグ）

- **顧客企業名＋個人名の組み合わせ**：`〇〇株式会社の△△さん` パターン
- **URL の api key 含み**：クエリパラメータに `key=` `token=` `secret=` を含む URL
- **金額（外部非公開）**：予算・給与・契約額を議事録本文に書かない。必要な場合は `status/releases.md` の内部ドキュメントに移す

### 低リスク（記載可能）

- 公開済み URL（クエリ無し）
- 会社名単独（個人名とのセットでない）
- 公式に発表されているリリース情報
